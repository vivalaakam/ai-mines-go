# Idle Mining Game — спецификация проекта на Go + Ebitengine + Lua

## 1. Общая концепция

Игра является idle mining / exploration проектом, вдохновлённым Deep Corp и Gold & Goblins.

Игрок исследует подземные уровни, добывает ресурсы, нанимает и объединяет работников, выполняет заказы, расширяет склады и постепенно спускается глубже.

Каждый уровень представляет собой потенциально бесконечную процедурную карту, генерируемую по seed-фразе чанками. Игрок начинает в пустой области 3×3 клетки, постепенно разрабатывает соседние клетки и ищет пустую область спуска 3×3, ведущую на следующий уровень.

Ключевое техническое ограничение проекта:

> Вся игровая механика, расчёты, правила, валидация команд и изменение состояния должны выполняться в Lua-слое. Go + Ebitengine отвечают за окно, игровой цикл, ввод, отрисовку, загрузку Lua, вызов Lua API и сохранение через адаптеры.

## 2. Главный архитектурный принцип

Проект должен быть разделён минимум на четыре слоя:

1. **Go Ebitengine App**

    * окно;
    * основной game loop;
    * input;
    * camera;
    * вызовы Lua API;
    * передача команд в Lua;
    * получение view-model из Lua;
    * вызов persistence adapter;
    * отрисовка через Ebitengine.

2. **Go Render / UI Layer**

    * визуализация карты;
    * визуализация работников;
    * визуализация складов;
    * визуализация заказов;
    * панели планирования смены;
    * hover/selection state;
    * camera state;
    * sprite/tile atlas;
    * анимации.

3. **Lua Game Engine**

    * вся доменная логика;
    * генерация уровней и чанков;
    * состояние клеток;
    * добыча ресурсов;
    * расчёт работников;
    * склады;
    * заказы;
    * смены;
    * tick-система;
    * валидация команд;
    * расчёт достижимости;
    * расчёт видимости;
    * подготовка данных для чтения.

4. **Go SQLite Persistence Adapter**

    * структурное сохранение;
    * загрузка сохранения;
    * восстановление Lua engine state;
    * создание Lua runtime с загруженным состоянием;
    * сохранение актуального состояния из Lua в SQLite.

Go-код не имеет права реализовывать игровые правила. Go может только:

* передать команду в Lua;
* получить результат;
* отрисовать результат;
* сохранить или загрузить состояние;
* выполнить техническую инфраструктуру.

## 3. Единственный API взаимодействия с Lua engine

Взаимодействие Go-слоя с Lua engine разрешено только через два метода:

```lua
engine.apply(command)
engine.read(query)
```

`apply(command)`:

* меняет состояние игры;
* валидирует команду;
* возвращает результат;
* возвращает события;
* может вернуть ошибку.

`read(query)`:

* только читает состояние;
* не меняет время;
* не генерирует игровую логику;
* не меняет карту;
* не двигает работников;
* не обновляет заказы.

Любой обход этого API запрещён.

Go-слой не имеет права напрямую менять:

* карту;
* клетки;
* работников;
* заказы;
* склады;
* деньги;
* смены;
* видимость;
* достижимость;
* progress клеток;
* состояние генератора.

## 4. Роль Ebitengine

Ebitengine используется как 2D-движок для Go-приложения.

Ebitengine отвечает за:

* создание окна;
* основной loop;
* вызов `Update`;
* вызов `Draw`;
* input;
* отрисовку tile-map;
* отрисовку UI;
* отрисовку работников;
* отрисовку эффектов;
* camera movement;
* масштабирование;
* обработку кликов.

Ebitengine не должен содержать игровую механику.

`Update` в Go не должен сам рассчитывать добычу, склад, заказы или работников. Вместо этого он должен вызывать Lua:

```go
result := luaEngine.Apply(Command{
    Type: "tick",
    Payload: map[string]any{
        "ticksPassed": 1,
    },
})
```

или:

```go
result := luaEngine.Apply(Command{
    Type: "assign_worker_to_target_cell",
    Payload: payload,
})
```

`Draw` должен использовать только данные, полученные через `read`.

## 5. Lua как авторитетный слой механик

Lua engine является единственным источником истины по игровому состоянию.

В Lua должны находиться:

* модели состояния;
* команды;
* запросы;
* генератор карты;
* расчёт tick;
* расчёт смен;
* расчёт заказов;
* расчёт складов;
* расчёт работников;
* расчёт добычи;
* расчёт merge;
* расчёт покупки;
* расчёт видимости;
* расчёт доступности;
* расчёт pathfinding/flood fill;
* расчёт автосейв-событий;
* подготовка state snapshot для сохранения.

Go может предоставлять Lua технические host-функции, но эти функции не должны содержать игровую логику.

Допустимые host-функции:

* логирование;
* чтение времени только для UI/debug, но не для game simulation;
* доступ к random seed helper, если это нужно;
* сериализация/десериализация;
* SQLite adapter bridge;
* panic/error boundary;
* performance profiling.

Недопустимые host-функции:

* `calculateMining`;
* `completeOrder`;
* `generateCell`;
* `mergeWorkers`;
* `calculateStorage`;
* `unlockLevel`;
* `updateVisibility`;
* `validateAssignment`.

Такие функции должны быть реализованы в Lua.

## 6. Модель времени

Игра использует дискретное игровое время.

* 1 игровой tick = 1 секунда.
* Offline-прогресс для MVP не нужен.
* Игровое время движется только через Lua-команду `tick`.

Команда времени:

```lua
engine.apply({
  type = "tick",
  ticksPassed = 1
})
```

Игровое время непрерывно: нет границ смен, нет фаз и нет `remainingTicks`.
Команда `tick` просто продвигает `state.gameTime.tick` на `ticksPassed` и на
каждом пройденном тике прогоняет логику добычи и расчётов по заказам.

Работники добывают, клетки разрабатываются, склады заполняются, заказы
получают ресурсы по мере их поступления — всё это происходит непрерывно, без
разделения на фазы.

Покупка работников, merge, покупка и улучшение складов, принятие/отклонение
заказов, выставление приоритетов заказов и (пере)назначение работников
разрешены в любой момент времени — блокировок по фазам больше нет.

## 8. Карта и чанковая генерация

Карта уровня бесконечная и лениво генерируемая по чанкам.

* Размер чанка для MVP: 32×32 клетки.
* Уровень генерируется по seed-фразе.
* При создании уровня сразу генерируется область 5×5 чанков.
* В центральном чанке находится стартовая область 3×3.
* В одном из нецентральных чанков области 5×5 находится спуск 3×3.
* Спуск не виден сразу.
* Спуск становится видимым только при попадании в радиус разведки.
* К спуску нужно пробуриться.

Генератор обязан гарантировать потенциальную достижимость спуска от стартовой области.

Идентификатор генерации чанка должен зависеть от:

```txt
seedPhrase
levelDepth
chunkX
chunkY
generatorVersion
```

Генерация должна быть детерминированной. Один и тот же seed и одна и та же версия генератора должны давать одинаковые чанки.

## 9. Старт и спуск

Каждый уровень имеет стартовую область:

```txt
3×3 empty cells
```

Стартовая область:

* находится в центральном чанке стартовой зоны 5×5;
* уже открыта;
* проходима;
* достижима;
* является базовой точкой для расчёта расстояния.

Каждый уровень имеет область спуска:

```txt
3×3 stairs_area cells
```

Область спуска:

* пустая;
* не содержит ресурсов;
* не содержит породы;
* находится в одном из нецентральных чанков области 5×5;
* скрыта до попадания в радиус разведки;
* открывает следующий уровень, когда до неё появляется достижимый путь.

## 10. Типы клеток

```lua
CellKind = {
  "empty",
  "deposit",
  "obstacle",
  "stairs_area"
}
```

### empty

Пустая проходимая клетка.

Не требует разработки.

### deposit

Клетка с породой и/или ресурсами.

После полной выработки всех компонентов становится `empty`.

### obstacle

Неразрушаемое препятствие.

Не может быть разработано.

Не становится проходом.

### stairs_area

Пустая клетка зоны спуска.

Зона спуска всегда имеет размер 3×3.

## 11. Видимость и достижимость

Нужно разделять видимость и доступность.

```lua
CellVisibility = {
  "unknown",
  "scouted"
}

CellAccessibility = {
  "unreachable",
  "reachable"
}
```

`scouted` означает, что игрок видит клетку и её состав.

`reachable` означает, что до клетки есть путь от входа уровня.

Игрок видит вглубь на 5 клеток от каждой открытой/достижимой клетки.

После каждого открытия новой клетки Lua engine должен:

1. пересчитать достижимую область;
2. открыть все связанные пустоты;
3. догенерировать нужные чанки;
4. обновить радиус разведки 5 клеток от всех достижимых клеток.

Если пустая область уходит за границу чанка, Lua engine должен запросить генерацию соседнего чанка. Если после этого область снова уходит за границу, процесс повторяется.

Генератор должен стараться избегать больших пещер. Большие связные пустые области нежелательны.

## 12. Пустоты

Пустоты являются проходимыми клетками.

Если игрок пробурился к пустой области, вся связанная пустая область становится доступной сразу.

После этого от каждой клетки этой пустой области считается радиус разведки 5 клеток.

Пустоты:

* не содержат компонентов;
* не дают ресурсов;
* не требуют разработки;
* могут использоваться как позиции для работников;
* могут открывать видимость дальше.

## 13. Неразрушаемые препятствия

Неразрушаемые препятствия разрешены.

Они:

* не добываются;
* не исчезают;
* не становятся проходом;
* могут блокировать путь.

Генератор обязан гарантировать, что от стартовой области до зоны спуска существует путь через пустые и потенциально разрабатываемые клетки.

После генерации стартовой зоны 5×5 чанков Lua engine должен выполнить validation:

```lua
GenerationValidation = {
  hasPathFromEntranceToExit = true,
  corridorWidthLimitRespected = true,
  stairsAreaReachablePotentially = true
}
```

Если проверка не пройдена, генератор должен детерминированно исправить уровень:

* пробить гарантированный маршрут;
* перестроить локальный участок;
* перегенерировать проблемную область через deterministic retry index.

## 14. Коридоры и ресурсные зоны

Уровень состоит из:

* ресурсных зон;
* породы;
* пустот;
* коридоров;
* неразрушаемых препятствий;
* стартовой области;
* зоны спуска.

Коридоры между источниками ресурсов не должны быть шире 3 клеток.

Ресурсная зона может быть шире 3 клеток.

Требование:

> Коридоры между ресурсными зонами не могут быть шире 3 клеток. Это ограничение не применяется к самим ресурсным зонам.

## 15. Компоненты клетки

Клетка `deposit` состоит из компонентов.

```lua
CellComponent = {
  type = "resource" или "rock",
  resourceId = nil или ResourceId,
  initialAmount = 0,
  remainingAmount = 0,
  ratio = 0
}
```

Порода является компонентом клетки, но не попадает на склад.

Пример клетки:

```txt
rock: 60%
stone: 25%
copper: 10%
gold: 5%
```

Работники вырабатывают компоненты равномерно.

Если за тик добывается 10 единиц работы:

```txt
rock -6
stone +2.5
copper +1
gold +0.5
```

`rock` исчезает.

`stone`, `copper`, `gold` попадают на склады, если там есть место.

## 16. Переполнение склада при смешанной клетке

Если склад для конкретного ресурса заполнен, этот ресурс временно не добывается.

Он:

* не теряется;
* остаётся в клетке;
* может быть добыт позже.

Остальные доступные компоненты продолжают вырабатываться.

Если в клетке есть:

* порода;
* медь;
* золото;

и склад золота заполнен, но для меди есть место, а порода ещё осталась, то работники продолжают:

* разбирать породу;
* добывать медь;
* не трогать золото.

Если в клетке остались только ресурсы, но все соответствующие склады заполнены, работники простаивают.

Состояние работников на такой клетке:

```lua
WorkerState = "blocked_by_storage"
```

Клетка считается полностью выработанной только когда все компоненты имеют:

```lua
remainingAmount = 0
```

После этого клетка становится пустой и проходимой.

## 17. Работники

Работники являются глобальным пулом игрока.

Любого свободного работника можно назначить на любой доступный уровень.

Работник имеет:

* id;
* уровень;
* скорость;
* состояние;
* текущий уровень назначения;
* целевую клетку;
* позицию, на которой он стоит;
* режим задания.

```lua
Worker = {
  id = "worker_1",
  level = 1,
  speed = 1.0,
  state = "idle",
  assignedLevelId = nil,
  targetCellId = nil,
  positionCellId = nil,
  assignmentMode = nil
}
```

Состояния:

```lua
WorkerState = {
  "idle",
  "working",
  "blocked_by_storage"
}
```

## 18. Merge работников

Merge происходит по правилу:

```txt
2 одинаковых работника → 1 работник следующего уровня
```

Примеры:

```txt
2 × Worker Lv1 → 1 × Worker Lv2
2 × Worker Lv2 → 1 × Worker Lv3
2 × Worker Lv3 → 1 × Worker Lv4
```

Merge возможен только для свободных работников.

Нельзя объединять работников, которые:

* работают;
* заблокированы складом;
* назначены на клетку;
* находятся в активном задании.

## 19. Покупка работников

Работники покупаются за деньги.

Сначала можно покупать только работников 1 уровня.

Доступный для покупки уровень зависит от максимального достигнутого уровня работника.

Правило:

```lua
maxPurchasableWorkerLevel = math.max(1, highestUnlockedWorkerLevel - 2)
```

Примеры:

```txt
Максимальный работник 1 уровня → можно покупать 1 уровень
Максимальный работник 2 уровня → можно покупать 1 уровень
Максимальный работник 3 уровня → можно покупать 1 уровень
Максимальный работник 4 уровня → можно покупать 2 уровень
Максимальный работник 5 уровня → можно покупать 3 уровень
Максимальный работник 6 уровня → можно покупать 4 уровень
```

Покупка работников доступна только в фазе планирования смены.

## 20. Разработка клеток

Игрок выбирает соседнюю закрытую клетку для разработки.

Работник не стоит на закрытой клетке.

Работник стоит на одной из соседних открытых достижимых клеток:

* сверху;
* снизу;
* слева;
* справа.

Если вокруг целевой клетки есть несколько свободных открытых достижимых клеток, на одну целевую клетку можно назначить несколько работников.

Максимум:

```txt
4 работника на одну целевую клетку
```

По одному с каждой стороны.

Одна открытая клетка может быть занята только одним работником.

Скорость разработки целевой клетки равна сумме скоростей всех работников, назначенных на неё.

Индивидуальный вклад работников не отслеживается.

Важно только:

* общий прогресс;
* общий объём добытого;
* общее состояние клетки.

## 21. Перемещение работников

Перемещение работников мгновенное.

Но работник может быть поставлен только на клетку, до которой есть путь от входа уровня.

Работник не может телепортироваться в закрытую или изолированную область.

Lua engine при назначении обязан проверить:

1. целевая клетка доступна для разработки;
2. позиция работника находится рядом с целевой клеткой;
3. позиция работника открыта;
4. позиция работника достижима от входа уровня;
5. позиция работника не занята другим работником;
6. работник свободен;
7. целевая клетка ещё не полностью выработана.

Команда:

```lua
engine.apply({
  type = "assign_worker_to_target_cell",
  workerId = "worker_1",
  levelId = "level_1",
  targetCellId = "cell_10_15",
  positionCellId = "cell_10_14",
  assignmentMode = "until_completed"
})
```

## 22. Назначения работников

Поддерживаются два режима:

```lua
AssignmentMode = {
  "shift_task",
  "until_completed"
}
```

`shift_task`:

* унаследованное от старой модели смен значение; принимается как валидное, но не несёт особого поведения — это поле-заглушка в модели данных.

`until_completed`:

* работник продолжает работать над клеткой, пока она не будет полностью выработана.

(Пере)назначение работников разрешено в любой момент — фаз, ограничивающих это, больше нет.

## 23. Склад

Для каждого ресурса есть отдельные склады.

Один склад хранит один тип ресурса.

Для одного ресурса может быть несколько складов.

Вместимость складов одного ресурса суммируется.

Склады можно:

* покупать;
* улучшать.

```lua
Storage = {
  id = "storage_1",
  resourceId = "stone",
  level = 1,
  capacity = 100,
  storedAmount = 35
}
```

Если для ресурса нет свободной вместимости, этот ресурс не добывается.

Покупка и улучшение складов доступны только в фазе планирования смены.

## 24. Деньги

Деньги являются отдельной валютой.

Деньги используются для:

* покупки работников;
* покупки складов;
* улучшения складов.

Деньги получаются через выполнение заказов.

Прямая продажа ресурсов без заказа для MVP не требуется.

## 25. Заказы

Продажа ресурсов происходит через систему заказов.

Заказ может требовать:

* один ресурс;
* несколько ресурсов.

```lua
Order = {
  id = "order_1",
  state = "available",
  rewardMoney = 100,
  expiresAtTick = 1500,
  acceptedAtTick = nil,
  priority = 1,
  requirements = {
    {
      resourceId = "stone",
      requiredAmount = 50,
      deliveredAmount = 0
    },
    {
      resourceId = "copper",
      requiredAmount = 10,
      deliveredAmount = 0
    }
  }
}
```

Состояния заказа:

```lua
OrderState = {
  "available",
  "accepted",
  "completed",
  "expired",
  "declined"
}
```

Игрок может отклонить заказ, если не собирается его выполнять.

Если игрок начал выполнение заказа, заказ нельзя отменить.

Если ресурсов хватает полностью, заказ выполняется сразу.

Если ресурсов не хватает, игрок может принять заказ в работу. После этого заказ становится обязательным.

В конце каждой смены Lua engine распределяет доступные ресурсы между активными заказами.

## 26. Распределение ресурсов между заказами

Если несколько заказов претендуют на один ресурс, engine должен поддерживать режим распределения.

```lua
OrderAllocationMode = {
  "priority_based",
  "proportional"
}
```

Для MVP предпочтителен:

```lua
priority_based
```

В этом режиме сначала закрываются заказы с более высоким приоритетом.

Приоритеты заказов можно менять в фазе планирования.

## 27. Ресурсы и глубина

Максимум ресурсов для первой версии: 12.

Ресурсы имеют:

* id;
* название;
* редкость;
* базовую цену;
* минимальную глубину появления;
* вес генерации по глубине;
* базовую вместимость склада.

```lua
ResourceConfig = {
  id = "copper",
  name = "Copper",
  rarity = "uncommon",
  unlockDepth = 2,
  basePrice = 5,
  storageBaseCapacity = 100,
  generationWeightByDepth = {
    { fromDepth = 1, weight = 0 },
    { fromDepth = 2, weight = 10 },
    { fromDepth = 5, weight = 25 }
  }
}
```

Чем глубже уровень, тем выше вероятность встретить большее разнообразие ресурсов.

Каждый новый уровень гарантированно добавляет минимум один новый ресурс в пул генерации, пока не будут открыты все 12 ресурсов.

## 28. Сохранение

Для MVP используется SQLite.

Запрещено сохранять всё состояние как один JSON blob.

SQLite должен хранить данные структурно.

Минимальные таблицы:

```sql
saves
levels
chunks
cells
cell_components
workers
storages
orders
order_requirements
```

Сохранение должно включать:

* seed-фразу;
* версию генератора;
* версию схемы;
* игровое время;
* текущую фазу;
* текущую смену;
* деньги;
* открытые уровни;
* сгенерированные чанки;
* все клетки;
* компоненты клеток;
* состояние видимости;
* состояние достижимости;
* работников;
* склады;
* заказы;
* открытые ресурсы;
* настройки правил.

SQLite adapter реализуется на Go.

Но SQLite adapter не должен содержать игровую логику.

Он только:

* читает таблицы;
* собирает Lua-compatible state;
* создаёт Lua engine с этим state;
* получает snapshot из Lua;
* сохраняет snapshot в таблицы.

## 29. Persistence adapter

Persistence adapter является отдельным слоем.

Он должен уметь:

```go
LoadEngine(saveID string) (*LuaEngine, error)
SaveEngine(engine *LuaEngine) error
CreateNewEngine(config NewGameConfig) (*LuaEngine, error)
```

Фактические имена могут отличаться, но смысл должен сохраняться.

Go app не должен напрямую читать SQLite для отображения игрового состояния.

UI получает данные только через:

```lua
engine.read(query)
```

## 30. Автосейвы

Сохранение выполняется:

* вручную по требованию игрока;
* автоматически после каждой смены.

Lua engine не пишет в SQLite сам.

Lua больше не генерирует события автосохранения. Вместо этого Go app layer
(`internal/app/update.go`) ведёт периодический счётчик тиков и сам вызывает
autosave каждые `AutosaveIntervalTicks` тиков (см. `internal/app/game.go`),
независимо от того, что возвращает Lua.

## 32. Go layer requirements

Go layer отвечает за:

* запуск Ebitengine;
* создание окна;
* Ebitengine `Update`;
* Ebitengine `Draw`;
* input;
* camera;
* assets;
* tile rendering;
* UI rendering;
* Lua runtime;
* SQLite adapter;
* обработку событий от Lua;
* autosave orchestration.

Go layer не отвечает за:

* добычу;
* генерацию ресурсов;
* генерацию клеток;
* merge работников;
* выполнение заказов;
* расчёт складов;
* расчёт tick;
* расчёт смен;
* расчёт видимости;
* расчёт достижимости;
* правила покупки;
* правила назначения работников.

Если какая-то функция отвечает на вопрос “что должно произойти по правилам игры?”, она должна находиться в Lua.

## 33. Lua layer requirements

Lua layer должен содержать весь game engine.

Минимальный публичный Lua API:

```lua
engine = {}

function engine.apply(command)
  -- validate command
  -- mutate state
  -- return result
end

function engine.read(query)
  -- read only
  -- return view data
end

function engine.export_state()
  -- return serializable state for persistence
end

function engine.load_state(state)
  -- restore state from persistence
end
```

`read` не должен менять состояние.

`export_state` не должен менять состояние.

Все изменения состояния должны проходить через `apply`.

## 34. Ebitengine Update loop

Так как Ebitengine обновляется чаще, чем игровой tick, Go-слой должен иметь accumulator.

Игровой tick равен 1 секунде.

Ebitengine может работать с TPS 60, но Lua `tick` вызывается только когда накопилась 1 игровая секунда.

Пример логики:

```go
func (g *Game) Update() error {
    g.input.Handle()

    g.tickAccumulator++

    if g.tickAccumulator >= updatesPerGameTick {
        g.tickAccumulator = 0

        result := g.luaEngine.Apply(Command{
            Type: "tick",
            Payload: map[string]any{
                "ticksPassed": 1,
            },
        })

        g.handleLuaEvents(result.Events)
    }

    return nil
}
```

## 35. Render data

Go render layer не должен читать полный Lua state без необходимости.

Для карты нужно использовать query:

```lua
engine.read({
  type = "get_level_view",
  levelId = "level_1",
  viewport = {
    x = 0,
    y = 0,
    width = 40,
    height = 30
  }
})
```

Lua возвращает view-model:

```lua
{
  levelId = "level_1",
  cells = {
    {
      x = 10,
      y = 15,
      kind = "deposit",
      visibility = "scouted",
      accessibility = "unreachable",
      components = {
        { type = "rock", remainingAmount = 50 },
        { type = "resource", resourceId = "copper", remainingAmount = 10 }
      }
    }
  },
  workers = {},
  selected = nil
}
```

Go render layer превращает эти данные в sprites/tiles.

Минимальный набор команд:

```lua
-- time
{ type = "tick", ticksPassed = 1 }

-- workers

{ type = "merge_workers", workerIds = { "w1", "w2" } }
{ type = "assign_worker_to_target_cell", workerId = "...", levelId = "...", targetCellId = "...", positionCellId = "...", assignmentMode = "until_completed" }
```

Все команды валидируются в Lua.

Минимальный набор запросов:

```lua

{ type = "get_workers" }
{ type = "get_storage_state" }
{ type = "get_available_orders" }

{ type = "get_resources" }

```

Все запросы должны быть read-only.

## 38. Ошибки Lua engine

`apply` должен возвращать структурированный результат.

```lua
{
  ok = true,
  events = {},
  patch = {}
}
```

или:

```lua
{
  ok = false,
  error = {
    code = "worker_not_idle",
    message = "Worker is not idle",
    details = {}
  }
}
```

Go layer не должен парсить текст ошибки для логики.

Go layer должен использовать `error.code`.

## 39. Тестирование Lua механик

Так как вся механика находится в Lua, тесты механик должны запускать Lua engine без Ebitengine.

Обязательные тесты:

* генерация чанков по seed;
* одинаковый seed даёт одинаковую карту;
* стартовая область 3×3 создаётся корректно;
* спуск 3×3 создаётся в нецентральном чанке стартовой области 5×5;
* есть потенциальный путь от старта до спуска;
* видимость раскрывается на 5 клеток;
* пустоты раскрываются через flood fill;
* большие пустоты ограничиваются генератором;
* ресурсная клетка добывается пропорционально компонентам;
* заполненный склад блокирует только конкретный ресурс;
* ресурс не теряется при переполнении склада;
* клетка становится проходимой после полной выработки всех компонентов;
* несколько работников могут работать над одной целевой клеткой с разных сторон;
* нельзя поставить двух работников на одну позицию;
* merge работает по правилу 2 в 1;
* покупка работников учитывает максимальный достигнутый уровень;
* смена длится 300 ticks;
* fast-forward доходит только до конца текущей смены;
* заказы закрываются сразу при наличии ресурсов;
* активные заказы получают ресурсы в конце смены;
* autosave event появляется после завершения смены.

## 40. Тестирование Go слоя

Go-тесты должны проверять:

* Lua runtime создаётся;
* `apply` корректно вызывает Lua;
* `read` корректно вызывает Lua;
* ошибки Lua корректно преобразуются в Go-ошибки;
* Ebitengine app не содержит доменной логики;
* SQLite adapter сохраняет и загружает структурированные данные;
* сохранённый state после загрузки создаёт эквивалентный Lua engine;
* autosave event вызывает persistence adapter;
* render layer работает только с view-model.

## 42. Правила для кодовых агентов

В репозитории должны быть файлы:

```txt
AGENTS.md
docs/architecture.md
docs/game-design.md
docs/engine-api.md
docs/persistence.md
```

`AGENTS.md` должен фиксировать:

1. Вся игровая механика реализуется только в Lua.
2. Go + Ebitengine не содержат доменных правил.
3. Go вызывает Lua только через `apply` и `read`.
4. UI/render не имеет права менять состояние напрямую.
5. SQLite доступен только через persistence adapter.
6. Сохранение не должно быть одним JSON blob.
7. Любое изменение механики должно сопровождаться Lua-тестами.
8. Любое изменение Go integration должно сопровождаться Go-тестами.
9. Перед завершением задачи агент обязан запустить проверки.
10. Если проверки прошли, агент обязан сделать commit в текущую ветку.
11. Если проверки не прошли, commit запрещён.
12. Агент обязан явно указать, какие проверки запускались.
13. Агент не имеет права переносить механику из Lua в Go без отдельного архитектурного решения.
